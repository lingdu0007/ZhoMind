# Chat Interface Stability Report

日期：2026-04-14
范围：`POST /api/v1/chat`、`POST /api/v1/chat/stream`

## 已知限制

1. 当前接口层做了 10s 超时保护；超过阈值会返回统一错误结构。
2. 流式接口在客户端断开时会记录中断日志并停止输出，不尝试重试已发送的 token。
3. 检索结果为空时，服务会降级为“未检索到足够相关的知识片段，请补充更具体的问题或关键词。”
4. 该链路依赖后端 retrieval / rerank / LLM 可用性；上游不可用会统一映射为错误结构。

## 重试建议

1. 对 `INTERNAL_ERROR` / `RAG_UPSTREAM_ERROR` 类错误，客户端可做 1 次指数退避重试。
2. 对 `VALIDATION_ERROR`、`AUTH_*` 类错误，不建议重试，应修正输入或鉴权。
3. 对流式断流，建议客户端重新发起同一问题请求，不建议从中断位置恢复 token 级续流。

## 超时策略

1. 同步接口 `/chat`：默认 10 秒超时。
2. 流式接口 `/chat/stream`：默认 10 秒超时保护；若流内无首包，可视为链路异常。
3. 业务上建议将长文本问题拆分为更小的问题，减少检索与生成耗时。

## 回归关注点

- 长文本输入：应保持契约不变且无 5xx 激增。
- 空输入：应稳定返回 422 统一错误。
- 断流/取消：应保持日志与错误语义一致，不污染协议结构。
- 流式成功：`meta -> rag_step -> content -> trace -> done`。
